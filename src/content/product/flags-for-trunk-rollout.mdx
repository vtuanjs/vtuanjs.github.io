On an enterprise LMS I worked on — teachers running lessons, students submitting work — the contract forbade breaking the service, and it named the only windows a release could happen in. We still wanted to test in staging and preprod the day code was ready. So a merge could no longer mean a release. A feature flag is what splits the two apart. (A different system from the identity posts on this blog, which were built git-based.)

**A release should be reversible in seconds, and separate from the moment the code was written.**

This is part 4 of the series on our identity system. Part 3 covered how one person's access resolves across organizations; this part is the third piece: how a feature actually reaches production, and why merging code and releasing it to customers can't be the same event.

Trunk-based development needs this: engineering merges continuously, but the contract decides when anyone sees the result. (For the org-targeting side of flags, see the sibling post on organization-scoped feature flags; for the engineering mechanics, see [Trunk-based development: a coordination problem, not a Git problem](/management/trunk-based-development/).)

| | |
|---|---|
| **Problem** | Engineering wants to merge finished work daily, but a real feature takes weeks — and a contract only allows release at named windows. |
| **Why** | "Merged" and "released" get treated as the same event, forcing a choice between holding work back or releasing on a date nobody chose. |
| **Goal** | New work reaches production the day it's ready, dormant, and only becomes visible when someone deliberately decides it should. |

## Why it matters

**A contract that names the exact release windows makes the stakes explicit:** a break outside those terms isn't a bad week, it's a broken promise. There's no slack in the calendar to absorb a surprise found the night before a release. The only way to move fast without gambling on the release date is to split "tested in a production-like environment" from "visible to the customer" — two separate events, decided by two different people, at two different times.

## Ship it dark, release it on purpose

**A feature flag is a switch, not a deploy — flipping it decides who sees the change, not whether the code exists.**

While the flag is off, the code runs in production, dormant, next to everything else. Engineering integrates a change the day it's ready; the business decides, separately, when it goes live.

## The trade-off

| | Wait until finished, release to everyone | Ship dark, release on the contract window |
|---|---|---|
| Speed of integration | New work waits off to the side, sometimes for weeks | New work ships the day it's ready |
| Undoing a mistake | A new release, a wait, a public rollback | A switch flipped, in effect within minutes |
| What it costs | Integration pain, high-stakes launches | Discipline to remove flags once released |

**The cost lands on engineering, not the customer:** every flag is code that has to be maintained until someone actually removes it. Skip that discipline and the codebase fills up with switches nobody remembers the purpose of. That's a cost paid in cleanup, on your own schedule. The alternative — a broken contract discovered by the customer — isn't.
