Standing up a new environment with Terraform, everything is code — the network, the database, the service accounts — except the secrets. The database password, a few third-party API keys, a signing secret: none of that has a Terraform resource, so it ends up typed into the GCP console by hand, or dropped in a teammate's DM so the next person can paste it in. The rest of the environment is reproducible from `terraform apply`. The secrets are the one piece that isn't — a manual step hiding inside an otherwise fully declared environment.

| | |
|---|---|
| **Problem** | App secrets (DB password, API keys) have no Terraform resource, so they get created by hand outside the code that defines everything else about the environment. |
| **Why** | Plaintext secrets can't be committed, so the natural response is to push them out of the repo entirely — into a console click or a chat message. |
| **Goal** | A secret is a file in the same repo as the Terraform config, reviewed in the same PR, decryptable only by whoever Terraform has granted access. |

## The naive answer

Reach for a hosted secret manager and stop thinking about it — store the value there, have the app fetch it at boot. It solves *storage*, but it pulls secrets out of the Terraform-managed repo entirely, so they stop being reviewed alongside the infrastructure change that needs them and start living in a separate console with its own access model.

Neither path treats the secret as something Terraform provisions and grants access to. Both leave it as a side channel next to the infrastructure-as-code, not part of it.

## The real problem

The team already had a rule: nothing gets deployed by hand, everything goes through Terraform, reviewed in a PR. Secrets broke that rule by necessity, because a plaintext value can't sit in a `.tf` file. The actual requirement wasn't "store secrets somewhere secure" — a secret manager already does that — it was "let a secret be defined, reviewed, and provisioned the same way as the database or the service account next to it." That means the secret's *ciphertext* needs to be a file in the repo, and *who can decrypt it* needs to be a grant that Terraform itself creates, not a manual IAM click that drifts out of sync with the code.

## My answer

Secrets are encrypted with [SOPS](https://github.com/getsops/sops) against a GCP KMS key, and committed as ciphertext next to the Terraform module that consumes them. Terraform creates the KMS keyring and key, and grants `roles/cloudkms.cryptoKeyDecrypter` on it to the developers and CI service accounts who need it — the same `terraform apply` that provisions the database also provisions who is allowed to read its password. Decryption doesn't depend on a recipient list baked into the file, the way public-key schemes like age work — SOPS calls out to KMS to unwrap the data key, and GCP's IAM policy decides whether that call succeeds. Access is a policy, not an embedded key list.

```yaml
# secrets.enc.yaml — committed next to the Terraform module
db_password: ENC[AES256_GCM,data:Yk2f...,tag:...,type:str]
api_token:   ENC[AES256_GCM,data:9Qp1...,tag:...,type:str]
sops:
  kms:
    - arn: projects/my-project/locations/global/keyRings/app/cryptoKeys/secrets
```

```hcl
# grant decrypt access as part of the same apply that creates the DB
resource "google_kms_crypto_key_iam_member" "dev_decrypt" {
  crypto_key_id = google_kms_crypto_key.secrets.id
  role          = "roles/cloudkms.cryptoKeyDecrypter"
  member        = "group:developers@example.com"
}
```

Encryption stops a secret from being readable in the repo. It doesn't stop someone from staging a plaintext `.env` by mistake — that needs a second, independent control: a pre-commit hook scanning staged content for credential patterns before the commit lands.

```bash
# pre-commit: block staged plaintext secrets
git diff --cached -U0 | grep -E '(AKIA[0-9A-Z]{16}|-----BEGIN .* PRIVATE KEY)' \
  && { echo "leak: plaintext secret staged"; exit 1; }
```

## Why not GCP Secret Manager directly

Secret Manager is the more obvious tool — it's built for exactly this, with versioning and audit logs out of the box. The reason it wasn't the answer here: it's a separate store, provisioned and read out-of-band from Terraform's own state. A value in Secret Manager doesn't show up as a diff in the PR that changes the database it belongs to — you'd add a `google_secret_manager_secret_version` resource for the *pointer*, but the value itself still has to get in there somehow, by hand or through a side script, breaking the same "everything is reviewed as code" property that motivated moving off manual console clicks in the first place. Encrypting with KMS and committing the ciphertext keeps the secret's actual content inside the reviewable diff — Secret Manager would only move where the plaintext hides, not eliminate the out-of-band step.

## The trade-off

| Approach | Reviewed in the infra PR | Revoke access | Where plaintext ever exists | Extra runtime dependency |
|---|---|---|---|---|
| GCP Secret Manager | no — value set out-of-band | remove IAM binding | Secret Manager, at rest | yes, a live fetch at boot |
| SOPS + KMS, committed | yes — ciphertext is in the diff | remove IAM binding, no re-encryption needed | briefly, wherever it's decrypted for use | no — decrypt is local/CI-side |
| Plaintext + `.gitignore` | no — never committed | n/a | your working directory, indefinitely | no |

Revocation is where GCP KMS earns its keep over a per-recipient scheme like age: because SOPS asks KMS to unwrap the data key rather than embedding it per public key, removing someone's `cryptoKeyDecrypter` grant revokes them immediately — no re-encrypting every file they used to have access to. The real cost is elsewhere: decrypted values still touch disk or memory at deploy time, however briefly, and that window is on you to keep narrow — mount into memory, don't write it to a path that could ever get staged.

## Failure modes worth knowing before you copy this

- **The scanner has false negatives.** Pattern and entropy checks catch high-signal formats like cloud access keys and PEM headers; they miss a short, low-entropy password pasted into a config file. It reduces risk, it does not eliminate it.
- **Decrypted-on-disk windows.** Whatever process decrypts a value for local use or deploy leaves it in memory or on disk for that duration — treat it as sensitive and never let it land on a path that could be staged.
- **The KMS key is a single point of trust.** Everyone who can call decrypt on that key can read every secret encrypted to it. Scope keys per environment (staging vs production) rather than one key for everything, so a compromised staging grant doesn't reach production secrets.
- **History is forever.** If plaintext ever lands in a commit, rewriting history doesn't undo a clone that already happened. The only honest response is rotating the credential — the KMS key protects the ciphertext, not a secret that already leaked once.
