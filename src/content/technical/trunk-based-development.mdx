On an LMS I worked on, we used feature branches: open one per feature, merge only when the feature was done. A branch would sit open for weeks and accumulate thousands of changed files. Two people paid for that, in different ways:

- **The manager had no visibility.** The system showed no change for weeks — no way to answer "did we actually do anything this sprint."
- **QA had no time.** A giant diff landed all at once, right before release — one sprint to catch bugs in code that had been changing invisibly for a month.

**Long-lived branches defer integration risk to the worst possible moment: right before release, in the largest possible batch.**

The fix isn't stricter branch hygiene — it's separating integration from release, so there's no reason to hold a branch open in the first place:

- Merge small changes to staging continuously, flag **on**.
- Let preprod/prod run with the flag **off**.

That split buys the time to test and fix bugs before anyone flips the switch.

**The core invariant: `develop` is always deployable. Whether something is released is controlled by flags, not branch state.**

| | |
|---|---|
| **Problem** | Integration deferred to merge time means conflicts, stale context, and large blast radius per deploy. |
| **Why** | Feature branches couple "code is integrated" to "code is done" — so nothing merges until the whole feature is finished, and integration risk and review size both grow with how long that takes. |
| **Goal** | Every merge to `develop` is tested, integrated, and safe to deploy. Release is a flag flip, not a deploy. |

## Branch strategy

- **One long-lived branch:** `develop`.
- Engineers branch off `develop` into **short-lived branches** — target hours to two days, never weeks.
- Merge back via PR. **The PR is a review, not a staging environment.**
- Branch naming is irrelevant to the model. What matters: nothing accumulates more than a few days of divergence from `develop`.

A branch open for a week is already carrying integration debt — it finds out about conflicts at merge time, and the author has lost the mental context to resolve them cleanly.

## CI pipeline on merge

Every merge to `develop` triggers a pipeline with these steps, in order:

1. **Auto-tag the commit** — deterministic version string from the commit timestamp or sequence. This is the artifact identifier for the rest of the pipeline.
2. **Detect changed apps** — diff against the previous tag; only apps with changed files get rebuilt. In a monorepo with 10 services, a PR touching one service rebuilds one service.
3. **Build affected apps** — produces the artifacts tagged in step 1.
4. **Auto-deploy to staging** — staging always reflects the latest `develop`. No manual trigger, no coordination, no "did you deploy yet?"

The production promotion is always manual — someone picks a tag from `develop` and triggers the deployment workflow. This is the deliberate gate between "integrated" and "released."

## Timestamp gate

The deployment workflow rejects any tag older than what's currently running in production:

```
if tag_timestamp < current_prod_tag_timestamp:
    abort_deploy()  # before touching infrastructure
```

**What it prevents:** someone picks a tag from three weeks ago, deploys it, and silently overwrites two weeks of production releases. The gate makes that physically impossible without an explicit override.

**Forward-only promotion:** rollbacks are done by reverting in `develop` and deploying the new commit — never by picking an old tag. That keeps the rollback visible in commit history instead of being a silent deploy-artifact operation.

## Feature flags

Feature flags are the mechanism that makes short branches viable for incomplete features. The pattern is straightforward:

```typescript
if (flags.isEnabled('new-checkout-flow', userId)) {
  return newCheckoutFlow(cart);
}
return legacyCheckoutFlow(cart);
```

The flag is the release control, not the branch. An engineer can merge a half-built checkout flow on day one, keep merging incremental changes daily, and users see nothing until the flag is flipped. No branch accumulates. No conflict builds up.

**Flag discipline has three rules:**

1. **Name flags for the feature, not the implementation.** `new-checkout-flow` not `use-stripe-v3`. The flag guards a user-facing behavior, and the name should match what product controls.
2. **Create a cleanup ticket at flag creation.** Flags are temporary by definition. A flag that shipped 6 months ago and is at 100% rollout is dead code with extra steps. The cleanup is trivial if you do it within a sprint of full rollout; it's archaeology if you do it a year later.
3. **Default to off.** A flag that defaults to on is an anti-flag — incomplete code is now the default path for anyone who hasn't explicitly opted into the flag infrastructure.

Flag accumulation is the main failure mode of this model. Teams adopt the flag discipline but skip the cleanup, and after a year the codebase has 40 flags, half of which are fully shipped and no longer guarded. Treat flag count as a metric worth monitoring.

## Hotfix flow

**Branch naming:** every successful deploy to preprod or production creates a hotfix branch named after the deployed tag:

| Environment | Branch pattern | GitHub Actions variable |
|---|---|---|
| Preprod | `hotfixes/preprod/[tag]` | `HOTFIX_BRANCH_PREPROD` |
| Prod | `hotfixes/prod/[tag]` | `HOTFIX_BRANCH_PROD` |

The branch name is derived from the tag's annotation message (`Branch: <branch-name>`). When a hotfix itself gets deployed, post-deploy reads that annotation and resolves back to the original branch — **the same branch is reused, not a new one per incident.** A new branch is only created when a fresh release from `develop` is deployed.

**Trigger rules** for pushes to `hotfixes/**`:

- `hotfixes/preprod/*` → build + auto-deploy to preprod
- `hotfixes/prod/*` → build only — production stays manual

**Finding the target branch:** run the `GetCurrentRelease` workflow action before cherry-picking. It reads `HOTFIX_BRANCH_PREPROD` / `HOTFIX_BRANCH_PROD` and outputs:

- the branch name
- the exact `checkout` command
- the exact `cherry-pick` command

The variable is the single source of truth — without this step there's no reliable way to know which hotfix branch to target.

**The fix always starts in `develop`, never the hotfix branch.** This is the invariant that cannot reverse under pressure.

If the fix goes to the hotfix branch first — the instinct under an incident — it exists only in the hotfix branch. When the next release from `develop` is promoted, the fix is overwritten. This is the most common hotfix failure mode, and it's hard to catch because the system looks correct immediately after the incident.

## Why not release branches

The alternative: cut a `release/2.4` branch at a milestone, stabilize it, deploy it. Looks structured. In practice it creates two codebases running in parallel:

- A fix lands in `release/2.4` → **may or may not** get backported to `develop`.
- Under incident pressure, backport is an afterthought.
- The bug reappears in `release/2.5` six weeks later.

The release branch wasn't protecting production — it was hiding integration debt and splitting the fix surface across two places.

**Tags on `develop` do the same job without the split:** one codebase, one fix surface, and the tag makes no claim about ongoing stability — it's just a pointer to a known-good commit.

## Cost table

| Benefit | Cost | Failure mode |
|---|---|---|
| Integration risk surfaces at merge time, not release time | Every merge must pass CI | Flaky tests slow everyone down; teams start ignoring red builds |
| Staging always reflects latest `develop` | Feature flags accumulate | Old flags never get cleaned up; dead code with guards |
| Production promotions are explicit tag choices | Timestamp gate must be maintained | Gate drift lets old tags through |
| Hotfix path is pre-created | Order reverses under pressure | Fix goes to production first and gets lost on the next develop release |

**Non-negotiable dependency: CI reliability.** A flaky 40-minute test suite is worse in this model than in any other — it's the gate every engineer hits multiple times a day. One engineer who bypasses CI because "the test is flaky anyway" breaks the model for everyone. Fix the test; don't route around it.

## The practical starting point

Smallest increment that demonstrates the model:

1. Take one feature currently on a long-lived branch.
2. Break it into the smallest mergeable piece.
3. Put the incomplete surface behind a flag.
4. Merge.
5. Track how long the next PR for that feature takes to merge and review.

The diff size and conflict surface will both be smaller. That observation is more persuasive than this document.
